WordPress powers 43% of the web. For most of that history, "good for SEO" meant fast loading, clean URLs, and a sitemap plugin. That's still true for Google. But AI answer engines — ChatGPT, Claude, Perplexity, Google AI Overviews — need something different. They read content, not crawl indexes. That changes what "well-optimised" means.

## How WordPress handles the AEO surface

WordPress renders pages at request time from a MySQL database. A typical production stack adds:
- Yoast SEO or Rank Math for meta tags and JSON-LD
- WP Super Cache or WP Rocket for caching the output
- A CDN in front (Cloudflare, Bunny) to avoid hammering the origin

The structured data (JSON-LD) is generated by the SEO plugin. The quality depends on the plugin's configuration and the post author filling in the right fields. A post without a manually set "Article type" and "Author" in Yoast emits incomplete schema.

`llms.txt` is not built into WordPress at all. A plugin exists ([llms-txt for WordPress](https://wordpress.org/plugins/llms-txt/)), but it's a third-party add-on that must be installed, configured, and updated separately.

Markdown twins (`/raw/<slug>.md`) — clean copies of content that LLMs can read without parsing HTML — are not a WordPress concept. The REST API (`/wp-json/wp/v2/posts`) exposes content as HTML-in-JSON, not clean markdown.

| AEO output | WordPress | Glint |
|---|---|---|
| `llms.txt` | Plugin, manual config | Built in, auto-generated |
| `/raw/*.md` twins | Not available | Built in, every post |
| JSON-LD (Article) | Plugin-dependent, optional fields | Built in, always complete |
| sitemap.xml | Plugin (Yoast/Rank Math) | Built in |
| RSS/Atom feed | Built in | Built in |
| Clean markdown source | REST API returns HTML | Source *is* markdown |

## How Glint handles the AEO surface

Glint is an Astro static site generator with a content contract (Zod-validated frontmatter) and AEO emitters baked into the build. Every `glint build` outputs the full AEO surface — `llms.txt`, `/raw/*.md` twins, JSON-LD, sitemap, RSS — with zero per-post configuration.

The reason this is possible: content is markdown files, so the `/raw` twin is just the source file served directly. JSON-LD fields are derived from the frontmatter contract, which `glint doctor` enforces before the post can merge. There's no optional config to forget.

The output is a static `dist/` directory — HTML + JSON + markdown files on a CDN. No database on the read path, no runtime to attack, no plugin update to miss.

## Where WordPress genuinely wins

This isn't a one-sided comparison:

- **Non-technical editors.** WordPress has a visual block editor. Glint's authoring model is markdown files + PR review. If a client needs to publish without touching a file system, WordPress (or a CMS layer on top of Glint) is the right answer.
- **Plugin ecosystem.** WooCommerce, membership plugins, form builders, advanced custom fields — WordPress has a mature ecosystem Glint doesn't replicate.
- **Media library.** WordPress's media manager handles upload, resizing, and attachment at the click of a button. Glint's media strategy is in-repo assets or an S3-compatible store, which requires more setup.

If your primary problem is "non-technical people need to publish blog posts with images," WordPress is the correct tool. Glint is for teams comfortable with git and a terminal.

## The AEO-specific verdict

For content that needs to be cited by AI answer engines, the key metric is: **how cleanly can an LLM read this post?**

WordPress: runtime-rendered HTML with nav, sidebar, footer, cookie banners, and plugin scripts around the content. The SEO plugin adds JSON-LD if configured. No clean markdown source.

Glint: static HTML with JSON-LD always present. `/raw/blog/<slug>.md` is the exact source markdown. `llms.txt` registers every post. The content the LLM reads is identical to what the author wrote.

This isn't a slight against WordPress — it was built for a different era. The AEO surface is an afterthought you bolt on. In Glint it's a compile output.

## Migrating from WordPress to Glint

If you have an existing WordPress site, `glint import wordpress` takes a WXR export (from WordPress Tools → Export) and converts posts to schema-valid markdown files, maps the frontmatter, and downloads images. The output is a Glint content directory ready to build.

```sh
glint import wordpress --file export.xml --dir ./content/blog
```

Slug preservation and `redirects.json` generation are part of the import — existing URLs stay valid.
